# 发现引擎搜索效率优化

**优先级**: P2
**来源**: 调研分析（策略产出速度）+ notes/discovery-run1 经验

## 问题

当前发现引擎搜索效率低下，严重制约策略产出速度：

1. **搜索空间过大**：RSI 策略 100800 组合，selector 每个候选需 ~15 次回测，耗时数小时
2. **无效搜索浪费算力**：batch_results.csv 中 90%+ 结果 ROI=0（因数据不完整或策略在该币种无效）
3. **无智能裁剪**：不区分"无数据"和"策略无效"，对数据不足的币种仍跑完整回测
4. **串行执行**：各币种/策略组合串行搜索，无并行化

要从 $222 增长到 $1 亿，需要持续发现 1000+ 新策略，当前效率远不够。

## 需求

### 1. 币种准入标准 + 数据预检

搜索前先检查币种是否值得搜索：
- **流动性门槛**：24h ADV < $1M 的币种跳过（小币种策略不可执行）
- **数据健康**：查询 data-engine 数据完整率（与 data-pipeline-health 需求协同），完整率 < 90% 的跳过
- **spread 检查**：orderbook 前 3 档 spread > 0.1% 的标记为"高滑点风险"
- 跳过的币种记录原因到 `records/discovery_skipped.jsonl`
- `shared/src/criteria.rs` 新增 `CoinCriteria` 结构定义币种准入标准

### 2. 搜索空间裁剪

- **粗筛阶段**：先用少量代表性参数（如 3 组）快速回测，ROI < 5% 的币种+策略类型组合直接跳过
- **精筛阶段**：仅对粗筛通过的组合展开全参数空间搜索
- 控制精筛阶段候选数 < 500（notes/discovery-run1 经验）

### 3. 并行搜索

- 多币种并行回测（利用多核 CPU）
- 各策略类型独立搜索流（可分配给不同 quant agent）
- 搜索状态持久化：中断后可从断点恢复

### 4. 搜索报告

每轮搜索结束后输出效率报告：
- 总组合数 / 跳过数 / 粗筛通过数 / 精筛通过数
- 各阶段耗时
- 每个通过准入的策略的关键指标

## 涉及文件

- `discovery/src/scanner.rs` — 数据预检 + 粗筛/精筛流程
- `discovery/src/generator.rs` — 搜索空间裁剪
- `discovery/src/evaluator.rs` — 并行化回测

## 验收标准

- [ ] 数据不完整的币种自动跳过
- [ ] 粗筛阶段 < 总搜索时间的 10%
- [ ] 精筛阶段候选数控制在 500 以内
- [ ] 搜索效率提升 5x+（对比当前全量搜索）
