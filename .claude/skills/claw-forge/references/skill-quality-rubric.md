# OpenClaw 技能质量评估标准

对齐 OpenClaw skill-evaluator 的 8 大类别，每类包含多个子项。

## 评分等级

| 分数 | 等级 | 说明 |
|------|------|------|
| 0 | 缺失 | 完全没有相关内容 |
| 1 | 不足 | 有尝试但问题严重 |
| 2 | 基本 | 能工作但有明显不足 |
| 3 | 良好 | 满足要求，少量改进空间 |
| 4 | 优秀 | 超出预期，可作为范例 |

## 8 大类别

### 1. 功能完整性 (Completeness) — 权重 20%

- SKILL.md 描述是否完整清晰
- 所有声明的功能是否都有对应实现
- 输入/输出是否明确定义
- 边界情况是否有处理说明

### 2. 可靠性 (Reliability) — 权重 15%

- 脚本是否有错误处理
- 失败时是否有清晰的错误信息
- 是否有重试机制（对于网络操作）
- 是否处理了常见异常情况

### 3. 性能/Token 成本 (Efficiency) — 权重 10%

- SKILL.md 是否简洁（避免冗余信息占用 context）
- 脚本执行是否高效
- LLM 调用是否必要且精简
- 是否避免了不必要的循环/递归

### 4. Agent 可用性 (Agent Usability) — 权重 20%

- description 是否包含清晰的触发条件
- 子命令是否有明确的输入/输出格式
- 脚本的 CLI 参数是否标准化（argparse）
- 错误码是否规范（成功 0，失败 1）

### 5. 人类可用性 (Human Usability) — 权重 10%

- 文档是否易读易懂
- 是否有使用示例
- 安装/配置步骤是否清晰
- 是否有常见问题解答

### 6. 安全性 (Security) — 权重 10%

- 是否避免了命令注入风险
- API Key 是否通过环境变量传递（非硬编码）
- 文件操作是否有路径校验
- 是否有输入验证

### 7. 可维护性 (Maintainability) — 权重 10%

- 代码结构是否清晰
- 是否有适当的注释
- 是否遵循项目编码规范
- 文件组织是否合理

### 8. Agent 特有 (Agent-Specific) — 权重 5%

- 是否支持无人值守执行
- 输出是否机器可解析（JSON 等）
- 是否可以作为 pipeline 的一环
- 是否支持幂等操作

## 总分计算

```
总分 = Σ (类别平均分 / 4 × 权重 × 100)
```

满分 100 分。

## 发布门槛

| 总分 | 建议 |
|------|------|
| 80+ | 可以发布 |
| 60-79 | 需要改进，标注已知限制后可发布 |
| < 60 | 需要大改，不建议发布 |
